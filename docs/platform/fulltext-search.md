# 全文检索平台设计文档

## 1. 设计概述

EMOP平台的全文检索系统采用**混合存储架构**，在保持业务数据一致性的同时，提供高性能的全文搜索能力。系统设计的核心思想是将全文检索作为业务数据的**派生视图**，通过异步同步机制确保搜索数据的最终一致性。

### 核心设计原则

- **业务数据权威性**：业务数据始终是权威源，全文索引为派生数据
- **最终一致性**：允许秒级同步延迟(<5秒)，确保系统高可用性
- **权限继承**：全文搜索结果必须遵循原始业务数据的权限控制
- **性能优化**：针对搜索场景优化的专用存储结构

## 2. 架构设计

### 2.1 混合存储架构

```
业务数据层 (OLTP)          搜索数据层 (OLAP)
┌─────────────────────┐    ┌─────────────────────┐
│ 业务表               │    │ 全文索引表           │
│ - 实时数据           │───▶│ - 搜索优化结构       │
│ - 事务一致性         │    │ - pg_bigm索引       │
│ - 权限控制(RLS)      │    │ - 异步同步           │
└─────────────────────┘    └─────────────────────┘
```

### 2.2 为什么选择混合存储

**技术选型考量**：
- **简化运维**：避免引入Elasticsearch等外部组件，降低系统复杂度
- **数据一致性**：同一数据库内的数据同步，减少数据不一致风险
- **权限集成**：直接利用PostgreSQL的RLS机制实现权限过滤
- **性能平衡**：在搜索性能和系统复杂度间找到最佳平衡点

**方案对比**：
- **直接查询业务表**：实时性好，但复杂查询性能差
- **Elasticsearch方案**：搜索性能优秀，但运维复杂，数据同步困难
- **混合存储方案**：平衡性能、复杂度和一致性的最优选择

## 3. 数据同步机制

### 3.1 触发器驱动的变更捕获

采用**触发器+任务队列**模式捕获业务数据变更：

```
业务数据变更 → 触发器 → 同步任务表 → 后台处理 → 更新索引
```

**设计优势**：
- **完整性**：捕获所有数据变更，无遗漏
- **解耦性**：业务操作与索引更新解耦，避免影响业务性能
- **可靠性**：任务队列确保变更最终被处理
- **可观测性**：任务状态可追踪，便于监控和调试

### 3.2 增量同步策略

**核心挑战：版本对象的索引管理**

PLM系统中的版本对象（ItemRevision）具有特殊的业务语义：
- 同一编码（code）可有多个版本
- 用户通常希望搜索到"逻辑对象"而非具体版本
- 版本升级时需要无缝切换索引指向

**解决方案：三层更新策略**

```sql
-- 1. 尝试更新现有object_id记录（内容更新）
UPDATE search_index SET content = ? WHERE object_id = ?

-- 2. 尝试更新相同code记录（版本升级）  
UPDATE search_index SET object_id = ? WHERE object_type = ? AND code = ?

-- 3. 插入新记录（全新对象）
INSERT INTO search_index (object_id, object_type, content, code) VALUES (...)
```

**业务语义保证**：
- 每个object_id只有一条索引记录
- 相同object_type+code组合指向最新版本的object_id
- 支持无code对象的正常索引

## 4. 权限集成设计

### 4.1 权限过滤策略

面临的核心挑战：**如何在保证搜索性能的同时实现权限控制**

**分层权限策略**：

```
无RLS表 → 直接查询索引表（高性能）
有RLS表 → JOIN原表过滤（权限安全）
混合场景 → 分别查询后UNION（兼顾性能与安全）
```

### 4.2 智能权限检测

系统自动检测表的RLS状态，动态选择查询策略：

**检测逻辑**：
```sql
SELECT relrowsecurity FROM pg_class WHERE relname = 'table_name'
```

**查询路由**：
- **无RLS表**：直接查询索引表，最高性能
- **有RLS表**：JOIN原表应用权限策略，确保安全
- **混合场景**：组合查询，平衡性能与安全

## 5. 搜索优化设计

### 5.1 索引结构设计

**多字段合并索引**：
```
search_content: 所有可搜索字段的合并内容
suggest_code: 精确匹配的编码字段  
suggest_name: 精确匹配的名称字段
```

**索引策略**：
- **GIN索引**：使用pg_bigm的gin_bigm_ops优化中文搜索
- **分字段索引**：支持精确建议和模糊搜索的不同需求
- **类型索引**：支持按对象类型过滤

### 5.2 查询优化技术

**关键词处理**：
- 空格分词，支持多关键词AND查询
- SQL注入防护，安全的参数化查询
- 大小写不敏感的ILIKE匹配

**性能优化**：
- LIMIT/OFFSET分页减少内存消耗
- 对象类型预过滤减少扫描范围
- 统计信息缓存减少重复计算

## 6. 可观测性设计

### 6.1 同步监控

**任务状态跟踪**：
```
PENDING → PROCESSING → COMPLETED/FAILED
```

**监控指标**：
- 待处理任务数量
- 同步延迟时间
- 失败任务统计
- 索引记录数量

### 6.2 查询日志

**可配置的查询日志**：
```sql
SELECT fulltext.set_query_log(true);  -- 开启调试日志
```

**日志内容**：
- 实际执行的SQL语句
- 权限检查路径选择
- 查询执行时间
- 结果统计信息

## 7. 扩展性考虑

### 7.1 字段配置化

**元数据驱动的字段配置**：
- 通过元数据配置可检索字段
- 动态添加新的可搜索字段

### 7.2 性能扩展

**水平扩展支持**：
- 索引表分区（按对象类型或时间）
- 读写分离（主库写入，从库搜索）

## 8. 运维管理

### 8.1 索引管理

**重建策略**：
- 单类型增量重建
- 全量重建支持
- 在线重建不影响搜索服务

**清理机制**：
- 自动清理过期同步任务
- 孤立索引记录清理
- 索引碎片整理

### 8.2 故障恢复

**数据一致性保证**：
- 同步任务幂等性设计
- 失败任务自动重试
- 数据校验和修复机制

## 9. 总结

EMOP的全文检索设计在**简单性**、**性能**和**一致性**之间找到了最佳平衡点。通过混合存储架构，既保证了业务数据的权威性和一致性，又提供了优秀的搜索性能。权限集成和可观测性设计确保了系统的安全性和可维护性。

这个设计特别适合企业级平台的需求：数据安全要求高、权限控制复杂、但又需要良好的搜索体验。随着业务发展，可以在这个稳固的基础上逐步演进到更高级的搜索能力。
