# 图查询平台设计文档

## 1. 设计概述

EMOP平台的图查询系统是专门为**OLAP分析场景**设计的图数据分析引擎。它与现有的OLTP关系查询服务（`AssociateRelationService`、`StructuralRelationService`）形成**职责分离**的互补架构，专注于复杂的多跳关系分析和基础图算法应用。

### 核心设计理念

- **OLAP专用**：专门为复杂关系分析和基础图算法设计，不处理OLTP事务，或某些明确延时影响的OLTP场景下使用
- **异步分析**：接受数据同步延迟，换取强大的图遍历分析能力
- **算法实用**：提供路径分析、连通性分析等实用图算法
- **场景驱动**：针对平台内对象的影响分析、追溯分析等典型场景优化

## 2. 架构设计

### 2.1 职责分离架构

```
OLTP层 (实时事务)              OLAP层 (分析计算)
┌─────────────────────────┐    ┌─────────────────────────┐
│ AssociateRelationService │    │ Graph Query (GQ) API    │
│ - 实时关系增删改查        │    │ - 复杂路径分析           │
│ - 删除保护检查           │    │ - 影响分析算法           │
│ - 版本规则解析           │    │ - 图遍历应用             │
│                         │    │ - 可视化数据构建         │
│ StructuralRelationService│    │                         │
│ - 结构关系加载           │───▶│ Apache AGE图数据库       │
│ - 一对一/一对多关系      │    │ - 异步同步(秒级延迟)     │
│ - 外键关系处理           │    │ - 从库查询               │
└─────────────────────────┘    └─────────────────────────┘
```

### 2.2 为什么采用职责分离

**OLTP vs OLAP的本质差异**：
- **OLTP需求**：实时性、一致性、简单查询、高并发
- **OLAP需求**：复杂分析、大数据处理、容忍延迟、计算密集

**业务场景分类**：
- **OLTP场景**：对象关系构建、对象删除保护检查(没有被引用才可被删除)、版本关系加载
- **OLAP场景**：BOM影响分析、质量追溯、供应链风险评估、关系可视化

**设计优势**：
- **性能隔离**：分析查询不影响业务事务性能
- **技术专用**：各自采用最适合的技术栈
- **扩展独立**：可独立扩展分析能力而不影响业务系统

## 3. 使用约束与限制

### 3.1 核心约束条件

**数据时效性约束**：
- **同步延迟**：数据同步秒级延迟
- **最终一致性**：非强一致性，不适合事务性操作
- **版本滞后**：可能查询到稍旧版本的关系数据

**查询位置约束**：
- **从库查询**：在从库节点执行，减少主库压力
- **读写分离**：只支持查询操作，不支持数据修改
- **性能隔离**：分析查询不影响业务事务性能

### 3.2 权限控制要求

**权限强制性**：
- **全员管控**：所有用户（非bypass）必须受权限管控
- **权限继承**：图查询结果必须遵循原始业务数据的权限策略
- **安全优先**：在分析能力与数据安全冲突时，优先保证数据安全

**权限实现机制**：
- **会话权限传递**：图查询继承用户当前会话的权限上下文
- **RLS策略应用**：对有权限控制的表强制应用行级安全策略
- **结果过滤**：确保分析结果只包含用户有权访问的数据

### 3.3 适用场景判断

**✅ 适合使用图查询的场景**：
- 多层级BOM影响分析（可接受秒级延迟）
- 质量问题追溯分析（历史数据分析）
- 供应链关系路径分析（基于路径遍历）
- 关系网络可视化（展示用途）
- 连通性分析（组件依赖分析）

**❌ 不适合使用图查询的场景**：
- 删除保护检查（需要实时数据）
- 工作流状态判断（业务逻辑决策）
- 权限验证相关查询（安全要求高）

## 4. 关系数据模型

### 4.1 EMOP关系类型系统

**结构关系 (Structural)**：
- **存储方式**：外键字段直接存储
- **特点**：强依赖，生命周期绑定
- **典型场景**：BOM父子关系、分类树父子关系
- **查询性能**：优秀，适合频繁访问

**关联关系 (Associate)**：
- **存储方式**：关系表独立存储
- **特点**：松耦合，独立生命周期
- **典型场景**：文档引用、供应商关系
- **扩展性**：支持关系属性，版本规则

## 5. 图数据库集成

### 5.1 Apache AGE选型考虑

**技术选型驱动因素**：
- **同数据库集成**：避免跨系统数据同步复杂性
- **标准Cypher支持**：降低学习成本，生态丰富
- **PostgreSQL兼容**：利用现有运维经验和工具
- **扩展性**：支持自定义函数和基础图算法

### 5.2 图数据同步策略

**异步同步架构**：
```
业务变更 → 变更捕获 → 同步队列 → 图数据更新
```

**权限感知同步**：
- **源数据权限保持**：同步过程保持原始数据的权限属性
- **权限元数据同步**：同步业务数据的同时同步权限控制信息
- **权限策略应用**：图查询时动态应用用户权限策略

**同步优化**：
- **增量同步**：只同步变更的关系
- **批量处理**：聚合变更减少同步频率
- **热点优化**：优先同步常用关系路径
- **版本感知**：正确处理版本对象的图结构
- **权限一致性**：确保图数据权限与原始数据权限一致

### 5.3 索引管理策略

`Apache AGE`默认不创建任何索引，需要显式创建索引以优化查询性能。系统采用自动化索引管理策略，通过pg cron定时任务`daily-graph-index-check`每日凌晨确保索引完整性。

- **顶点表索引体系**：
    - **Vertex ID索引**：`BTREE(id)` - AGE内部顶点唯一标识符，用于图遍历时的快速节点定位
    - **Properties GIN索引**：`GIN(properties)` - 支持MATCH子句中的复杂属性组合查询，如`MATCH (n:ItemRevision {id: 'xxx', status: 'active'})`
    - **业务ID索引**：`BTREE(agtype_access_operator(properties, '"id"'))` - 业务主键精确查询和WHERE子句过滤
    - **业务Code索引**：`BTREE(agtype_access_operator(properties, '"code"'))` - 编码字段的精确匹配、范围查询和排序操作

- **边表索引体系**：
    - **Edge ID索引**：`BTREE(id)` - 边的唯一标识符，用于边查询和更新操作
    - **Properties GIN索引**：`GIN(properties)` - 边属性的复杂查询支持
    - **Start ID索引**：`BTREE(start_id)` - 出边查询优化，支持"从某节点出发"的路径遍历
    - **End ID索引**：`BTREE(end_id)` - 入边查询优化，支持"到达某节点"的反向路径查询

- **索引设计原理**：
    - **查询场景适配**：
        - **MATCH子句查询**：优先使用GIN索引进行属性模式匹配，如`MATCH (n:Label {prop1: 'value1', prop2: 'value2'})`
        - **WHERE子句查询**：使用BTREE索引进行精确条件过滤，如`WHERE n.id = 'specific_value'`
        - **路径遍历查询**：start_id/end_id索引支持高效的图遍历算法
          - **排序和范围查询**：依赖BTREE索引的有序特性，如`ORDER BY n.code`或`WHERE n.id > 'REV001'`
    - **性能权衡**：GIN索引支持复杂查询但占用更多存储空间；BTREE索引查询速度快但仅支持单属性操作
    - **图算法优化**：`start_id`/`end_id`索引是路径分析、连通性分析等图算法的性能基础
:::warning Apache AGE索引限制
- Apache AGE在WHERE子句中使用索引存在已知问题，特定属性的BTREE索引可能无法被WHERE子句有效利用
- 优先在MATCH子句中使用属性过滤而非WHERE子句，以更好地利用GIN索引的模式匹配能力
:::
- **自动化管理机制**：
    - **检查范围**：基于`ag_catalog.ag_label`表动态发现所有顶点和边标签
    - **创建逻辑**：区分顶点表(kind='v')和边表(kind='e')，应用不同的索引策略
    - **定时维护**：`daily-graph-all-index-check` cron任务每日0点执行，检查并创建缺失索引


### 5.4 图算法集成

**原生算法支持**：
- **路径算法**：最短路径、所有路径查询
- **变长路径遍历**：支持多跳关系遍历和深度控制
- **连通组件**：识别图中的连通子图
- **基础图查询**：节点度数计算、邻居查询

**业务场景分析**：
- **BOM层级遍历**：多级BOM结构的完整遍历
- **依赖路径分析**：基于路径查询的依赖关系分析
- **影响范围计算**：通过图遍历计算变更影响范围
- **追溯路径查询**：质量问题的完整追溯路径

## 6. API设计思想

### 6.1 接口设计

**设计目标**：为复杂图遍历分析提供直观、强大的查询接口

```java
// 专注于路径分析场景的API设计
GQ.from(document)
  .out("references", "dependencies")  // 多类型关系遍历
  .depth(5)                          // 深层分析
  .paths()                          // 路径查询
  .as(Project.class)                // 结果类型约束
  .analyze();                       // 执行分析查询
```

**设计原则**：
- **遍历导向**：API设计围绕图遍历和路径分析场景
- **路径优化**：内置路径查询和优化能力
- **结果丰富**：支持图结构、路径数据、可视化数据等多种输出
- **性能感知**：提供查询复杂度估算和性能提示

### 6.2 与OLTP服务的协作模式

**协作而非替代**：
```java
// OLTP: 实时关系查询
List<Document> directRefs = associateRelationService
    .findAllChildren(part, RelationType.DOCUMENT_REFERENCE);

// OLAP: 深度路径分析  
GraphView<ModelObject> pathNetwork = GQ.from(part)
    .out("*")
    .depth(6) 
    .paths()
    .graph();
```

**分工协作模式**：
- **实时展示**：使用OLTP服务获取当前关系状态
- **路径分析**：使用图查询分析复杂路径和连通性
- **决策支持**：OLTP提供精确数据，图查询提供路径洞察
- **用户体验**：实时响应+深度分析的组合体验

## 7. 典型业务分析场景

### 7.1 产品结构深度分析

**BOM影响路径分析**：
- **分析需求**：某零件变更对产品结构的影响路径
- **分析深度**：5-10层的BOM结构遍历
- **算法应用**：多层路径遍历 + 连通性分析
- **时效要求**：秒级延迟可接受，重点在分析完整性
- **vs OLTP**：OLTP服务处理单层BOM查看，图查询处理深度遍历

**产品依赖关系网络**：
- **分析目标**：基于依赖关系识别产品模块连通性
- **关系挖掘**：零件依赖关系 + 产品系列关系
- **算法支持**：连通组件分析 + 路径查询
- **业务价值**：依赖关系可视化、模块化设计支持

### 7.2 供应链路径分析

**多级供应商路径追踪**：
- **分析场景**：追踪供应商到最终产品的完整路径
- **关系网络**：供应商 → 原材料 → 零件 → 产品 → 项目
- **算法模型**：最短路径 + 所有路径查询
- **决策支持**：供应链可视化、路径优化
- **vs OLTP**：OLTP查询直接供应商，图分析多级路径

**供应商连通性分析**：
- **分析目标**：发现供应商网络的连通特性
- **关系发现**：合作关系、依赖关系连通性
- **算法应用**：连通组件分析 + 邻居查询
- **战略价值**：供应商网络优化、风险点识别

### 7.3 质量追溯路径分析

**质量问题追溯路径**：
- **分析需求**：质量问题的完整追溯路径查询
- **追溯深度**：从成品到原材料的所有路径
- **关系类型**：制造关系、批次关系、检验关系
- **算法支持**：最短路径 + 所有路径查询
- **vs OLTP**：OLTP处理单次追溯查询，图分析识别所有路径

**影响范围路径评估**：
- **分析场景**：某批次原材料问题的影响路径
- **传播路径**：批次 → 零件 → 装配 → 产品 → 客户
- **算法模型**：变长路径遍历 + 深度控制
- **应急响应**：影响路径识别、范围确定

### 7.4 知识网络路径分析

**技术文档关联路径**：
- **构建目标**：企业技术文档的关联路径分析
- **关系挖掘**：文档引用、技术关联路径
- **算法应用**：路径查询 + 连通性分析
- **创新支持**：知识路径识别、文档关联发现

**跨项目技术路径分析**：
- **分析需求**：识别技术方案在项目间的传播路径
- **关系模式**：项目 → 技术方案 → 设计模式 → 知识资产
- **算法支持**：路径遍历 + 邻居查询
- **价值输出**：技术传播路径、复用机会识别

### 7.5 场景选择指南

**图分析适用判断**：
```
✅ 多层级路径分析（3+跳）
✅ 连通性和依赖分析  
✅ 影响路径和传播分析
✅ 历史路径和演进分析
✅ 网络连通性分析
✅ 可视化和路径展示

❌ 实时业务决策
❌ 事务性操作支持
❌ 用户界面实时展示
❌ 权限验证相关
❌ 工作流状态查询
```

**与OLTP服务协作模式**：
- **数据获取**：OLTP提供准确的当前状态
- **路径分析**：图查询提供路径和连通性洞察
- **决策支持**：结合实时数据和路径分析结果
- **用户体验**：即时响应 + 深度路径分析的组合

## 8. 性能优化策略

### 8.1 查询优化

**查询规划优化**：
- **谓词下推**：在图遍历早期应用过滤条件
- **路径修剪**：基于业务规则的无效路径剪枝
- **结果缓存**：热点查询路径的结果缓存
- **索引利用**：充分利用图索引和关系索引

**内存管理**：
- **流式处理**：大结果集的流式返回
- **分页支持**：避免大图结构的内存溢出
- **对象池**：复用图节点和边对象

### 8.2 数据分区策略

**图数据分区**：
- **业务分区**：按产品线、项目分区
- **热度分区**：热点关系放入内存
- **时间分区**：历史数据归档分区
- **类型分区**：不同关系类型的分离存储

**读写分离**：
- **写入优化**：主库专注于关系变更
- **读取优化**：从库支持复杂图分析
- **一致性控制**：可接受的读延迟换取查询性能

## 9. 设计权衡与演进路径

### 9.1 OLAP专用设计的权衡

**优势**：
- **路径分析强大**：专门为复杂路径分析优化，遍历能力强
- **性能隔离完全**：分析查询完全不影响业务事务性能
- **技术栈专用**：可采用最适合图遍历的技术和优化策略
- **扩展性强**：可独立扩展分析能力，不受业务系统约束

**权衡与限制**：
- **数据延迟**：异步同步带来的秒级延迟
- **场景受限**：不适用于实时决策和事务性操作
- **运维复杂度**：需要维护额外的图数据存储和同步机制

**设计决策理由**：
- 图分析场景天然容忍延迟（路径分析、连通性分析）
- 基础图算法已能满足核心业务需求
- 路径查询的计算密集特性需要性能隔离
- 企业级应用重视稳定性胜过功能完整性

## 10. 总结

EMOP图查询系统的设计体现了**专业化分工**的架构哲学，通过明确的OLAP定位，与现有OLTP关系服务形成**互补协作**的关系。系统专注于复杂图路径分析场景，为EMOP平台的深度数据洞察提供强大支持。

**路径分析能力增强**：
- 从单层关系查询升级到多层路径分析
- 从静态数据展示升级到动态连通性发现
- 从人工分析升级到算法驱动的路径洞察

**架构价值**：
- **性能隔离**：分析计算不影响业务事务性能
- **技术专用**：针对图路径分析场景的深度优化
- **职责清晰**：OLTP事务与OLAP分析的明确分工
- **渐进演进**：在稳固基础上逐步增强分析能力

这种设计确保了EMOP平台既能满足日常业务的实时性要求，又能为企业决策提供深度的路径分析能力。通过专业化的分工，系统在复杂度和性能之间找到了最佳平衡点，为制造企业的数字化转型提供了坚实的技术基础。
