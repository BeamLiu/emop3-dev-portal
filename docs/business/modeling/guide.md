# EMOP 业务建模基本概念

## 概述
EMOP平台提供了强大而灵活的业务建模能力。本文将介绍 EMOP 业务建模中的核心概念，帮助您更好地理解和使用平台的建模功能。

## 核心概念

### Schema (数据库实例)
Schema 是 EMOP 中最顶层的数据组织单元，用于对不同领域的数据模型进行分类管理。每个业务模型都必须属于某个 Schema。系统预置了以下几个主要的 Schema:

- common: 通用基础数据
- cad: CAD相关数据 
- bom: BOM结构相关数据
- document: 文档管理相关数据
- material: 物料相关数据
- auth: 权限相关数据
- project: 项目管理相关数据

### ModelObject (模型对象)
ModelObject 是 EMOP 中所有业务对象的基础接口，具有以下特点：

- 每个 ModelObject 实例都对应一个具体的业务对象
- 包含对象类型信息(objectType)
- 支持动态属性扩展
- 可以持久化到数据库
- 支持元数据定义



ℹ️举例来说：
>一个发动机部件就是典型的 ModelObject：
>-   它有基本属性（零件号、版本号、状态）
>-   可能需要动态添加新的认证属性（如欧标、美标认证信息）
>-   存储在数据库中被各个环节引用
>-   遵循产品数据规范定义
>
>同样地发动机零件、图纸等等这些都是ModelObject

### Object Type (对象类型)
Object Type 是业务对象的类型标识，大部分`API`及`DSL`都支持以下两种形式：

1. 完全限定名（Full Qualified Name）：
   - 形如：`io.emop.model.bom.BomLine`
   - 包含完整的包路径
   - 全局唯一
   - 如果Object Type的定义是从Java定义自动生成的，则名称与Java class的名称完全一致

2. 简单名称（Simple Name）：
   - 形如：`BomLine`
   - 不包含包路径
   - 条件：在系统中没有重名的 Object Type

### Metadata (元数据)
Metadata 用于定义业务对象的结构和行为规范，具有以下特点：

1. 支持继承机制：
   - 子类继承父类的所有属性和关系
   - 子类可以覆盖父类的属性定义
   - 子类可以添加新的属性和关系

2. 支持多语言：
   - 对象名称的多语言
   - 对象描述的多语言
   - 属性名称的多语言
   - 属性描述的多语言

3. 定义方式，见[模型定义](modeling)：
   - Java 注解方式
   - DSL 方式

ℹ️举例来说：
>以发动机零件的元数据定义为例：
>-   继承机制：曲轴零件继承自通用机械零件的所有属性
>-   多语言：零件名称、描述同时支持中英文
>-   属性约束：重量必须大于 0，材料必须从标准库选择

### Storage Class (存储类)
Storage Class 指定了业务对象在数据库中的实际存储形式，任意一条数据都会以这个类的实例存放到数据库或从数据库取出的对象都是这个存储类：

- 定义对象的物理存储结构
- 可以被多个业务对象共用
- 支持继承关系，例如子类型没有定义存储类的情况下会自动使用父类的存储类进行存储
- 决定了对象的序列化方式

ℹ️举例来说：
>-   图纸、BOM等业务都具有自己不同的Java的定义作为存储类
>-   不同类型的文档（CAD、PDF、Office）则可以共用Document存储类，然后再在元数据当中定义不同的扩展属性
>-   定制化类型可以扩展或者直接使用平台内置存储类

### Business Unique Key (业务唯一键)
业务唯一键用于确保业务数据的唯一性：

- 支持单字段或多字段组合
- 在同一业务范围内保证唯一性
- 支持继承特性
- 平台自动进行唯一性校验，唯一键在保存时不能为空

ℹ️举例来说：
>-   图纸号 + 版本号（组合唯一键）
>-   研发项目编号（项目管理域的唯一键）

### Relation(关系类型)

EMOP支持以下几种主要的对象关系：

1. 结构关系 `StructuralRelation`：
- 标记对象`ModelObject`之间的结构关系。用于表达对象间的固定从属关系，这种关系通过`外键`字段直接存储。 
- 仅支持一对一和一对多关系，不支持多对多关系。
- 结构关系的特点：
	- 表达对象之间的强从属关系，子对象依赖于父对象
	- 关系是对象定义的固定组成部分
	- 使用外键字段直接存储，不支持在关系上定义额外属性
	- 查询性能好，适合频繁访问的关系
	- 仅支持一对一和一对多关系，多对多关系应使用 `AssociateRelation`

2. 关联关系 `AssociateRelation`
  - 标记对象 `ModelObject`之间的关联关系。用于表达独立业务对象之间的动态关联,  这种关系通过单独表 `COMMON.Relation`进行存储。
  - 支持一对多，多对一和多对多关系，特别适用于需要在关系上定义属性的场景。
  - 关联关系的特点：
	- 双方都是独立的业务实体
	- 具有独立的生命周期
	- 关系可以动态建立和解除
	- 使用关系表存储，支持在关系上定义任意key-value属性
    - 支持根据版本规则进行查询

3. 内嵌关系 `EmbeddingRelation`：
   - 主对象`ModelObject`包含任意复杂对象，内嵌对象在主对象内部存储，也无法单独访问，简单理解就是主对象有一个复杂对象的属性
   - 内嵌对象与主对象同生命周期

ℹ️举例来说：
>1.   结构关系：
>    -   BOM 中的父子件关系
>    -   工艺路线中的工序关系
>    -   配置结构中的选配关系
>2.   关联关系：
>    -   产品与其相关文档的关联
>    -   零件与其适用项目的关联
>    -   物料与供应商的关联
>    -   BOM的精确引用、非精确引用、最新工作版本等规则动态构建BOM树
>3.   内嵌关系：
>    -   产品规格参数表
>    -   物料的质检标准项
>    -   工艺参数配置


### Trait (特征)

Trait 是 EMOP 中一种可组合的功能单元，它定义了一组相关的属性和行为，用于实现业务对象的特定能力。每个 Trait 都封装了特定的业务能力，可以被不同的业务对象复用。主要特点：

1.  组合机制：
    -   支持多个 Trait 组合
2.  内置 Traits：
    -   HierarchicalTrait：层级关系能力, 典型场景如BOM树, 分类树等
    -   Tabular：表格数据能力
    -   BusinessCodeTrait：具备编码属性的对象, 典型场景如物料编码，文档编码，项目编码等
3.  扩展性：
    -   支持自定义 Trait
    -   提供配置项定制行为

ℹ️举例来说：

> 1.  使用 Hierarchical Trait 实现组织架构：
> 
> -   部门树形结构
> -   自动维护层级路径
> -   支持递归查询
> -   防止循环引用
> 
> 2.  使用 Tabular Trait 实现检验表：
> 
> -   动态定义列
> -   数据验证规则
> -   计算列支持
> -   批量数据处理
> 
> 3.  组合多个 Traits： 
> ```java
>// 自动获得HierarchicalTrait、RevisionableTrait、ApprovableTrait这些特征的能力
>public class CustomizedTask extends AbstractModelObject {
>    
>    private final HierarchicalTrait<CustomizedTask> hierarchical;
>    
>    private final RevisionableTrait<CustomizedTask> revisionable;>
>	
>    private final ApprovableTrait<CustomizedTask> approvable;
>}
>```

### Attribute(属性类型)

EMOP支持多种属性类型：

1.  基本属性:
-   支持动态添加新属性
-   支持属性值的查询操作

2.  可扩展枚举属性:
-   支持可扩展的枚举值定义
-   支持多语言配置
-   可以添加额外的属性信息

3.  计算属性:
-   基于其他属性进行动态计算
-   支持配置复杂的计算规则

4.  关系属性:
-   用于定义对象之间的关系，见[关系类型](#relation-关系类型)

ℹ️举例来说：
>1.   基本属性：
>    -   产品规格（尺寸、重量）
>    -   物料编码、名称
>    -   版本号、状态
>2.   可扩展枚举属性：
>    -   产品生命周期状态（设计中、试产、量产、停产）
>    -   文档状态（编制、审核、发布、作废）
>    -   物料分类
>3.   计算属性：
>    -   BOM 的成本汇总
>    -   工时统计
>    -   物料需求量
>4.   关系属性：
>    -   BOM 结构关系
>    -   工艺路线
>    -   文档关联关系

### ItemRevision (版本对象)

EMOP针对需要版本控制的对象提供升版，变更历史，变更控制等能力，任何需要版本控制的业务对象（如产品、零件、图纸等）都应该继承自 ItemRevision。版本对象有以下特点：
- 包含基础编码`code`和版本号`revId`
- 同一个编码可以有多个不同版本
- 自动维护业务对象的版本历史
- 支持通过业务编码进行逻辑对象查询

ℹ️版本应用示例一：
> 发动机设计迭代，发动机总成（ENG-001）：
>- A版本：概念设计
>- B版本：优化冷却系统
>- C版本：提升密封性能
>
> **逻辑视图**：用户看到"发动机总成(ENG-001)"，根据版本规则显示对应版本

ℹ️版本应用示例二：
>装配工艺文档，装配规范（SOP-002）：
>- A版本：初始工艺流程
>- B版本：增加质量检查点
>- C版本：优化装配顺序
>
> **逻辑视图**：用户看到"装配规范(SOP-002)"，可选择查看任意版本

ℹ️版本应用示例三：
>变速箱组件升级:
>1.   原版本（A版）标准配置
>2.   发现齿轮箱异响问题
>3.   创建新版本（B版）进行设计改进
>    -   保留A版本作为历史记录
>    -   在B版本上进行设计更改
>    -   更改完成后发布B版本
>4.   新产品使用B版本生产
>5.   可随时查看A版本的历史信息
>
> **逻辑视图**：用户在界面看到"变速箱组件"，系统根据版本规则自动显示B版本内容


### LogicalModelObject (逻辑对象)
LogicalModelObject 是业务对象在逻辑层面的抽象表示，用于运行时根据版本规则动态解析到一个具体的版本对象。

#### 核心特点：
- **业务视角**：用户看到的是业务概念（如"发动机设计"）
- **版本规则驱动**：通过版本规则动态决定显示哪个具体版本
- **查询抽象**：支持基于业务编码的逻辑查询

#### 与ItemRevision的关系：
```
LogicalObject: "发动机设计" (WORKFLOW-004005)
├── ItemRevision: WORKFLOW-004005/A (物理版本)
├── ItemRevision: WORKFLOW-004005/B (物理版本)
└── ItemRevision: WORKFLOW-004005/C (物理版本)
```

#### 应用场景：
- **全文检索**：搜索"发动机"时返回逻辑对象，根据版本规则展示对应版本
- **文件夹视图**：文件夹显示逻辑对象列表，用户选择版本规则查看不同版本
- **BOM结构**：BOM树显示逻辑关系，根据版本规则动态构建具体版本的BOM
- **关系图谱**：基于逻辑对象构建业务关系图

ℹ️举例说明：
>用户在文件夹中看到"发动机设计文档"，这是一个LogicalObject
>- 当选择"最新版本"规则时，显示C版本的内容
>- 当选择"最新发布版"规则时，显示B版本的内容
>- 当选择"工作版本"规则时，显示A版本的内容

### RevisionRule(版本规则)

版本规则定义了从逻辑对象到具体版本的解析规则：

#### 内置版本规则：
- **LATEST**: 最新版本（按创建时间）
- **LATEST_RELEASED**: 最新发布版本
- **LATEST_WORKING**: 最新工作版本
- **PRECISE**: 精确版本（直接指定版本号）
- **TIME_POINT**: 时间点有效版本，指定时间后生效
- **SCRIPT**: 根据自定义脚本判定有效版本

#### 应用场景：
- **设计环境**：通常使用 LATEST_WORKING 查看正在开发的版本
- **生产环境**：通常使用 LATEST_RELEASED 查看正式发布的版本
- **历史追溯**：使用 PRECISE 查看特定时间点的版本

通过理解这些基本概念，您将能够更好地使用 EMOP 平台进行业务建模，创建灵活且高效的业务应用。
